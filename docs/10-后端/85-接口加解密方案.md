# 接口加解密方案

> 状态：✅ 已实现（15/15 任务完成） — 2026-07-11
>
> 基于 RSA + AES-GCM 混合加密的前后端接口数据加解密方案。

## 加密协议

```
请求加密：AES-256-GCM 随机密钥加密请求体 → RSA-OAEP 公钥加密 AES 密钥
响应加密：AES-256-GCM 随机密钥加密响应体 → RSA-OAEP 公钥加密 AES 密钥 → RSA 私钥签名
```

## 加密数据格式

```json
{
  "data": "AES-GCM 加密后的 Base64 密文",
  "key": "RSA-OAEP 加密后的 AES 密钥 (Base64)",
  "iv": "AES-GCM 初始化向量 (Hex)",
  "nonce": "随机数 (Base64)",
  "timestamp": 1234567890,
  "signature": "RSA 签名 (Base64)"
}
```

## 响应处理流程

```
Controller 返回业务数据
  → ResponseEncryptAdvice (@Order HIGHEST_PRECEDENCE)
    → AES-GCM 加密数据 → RSA 加密 AES 密钥 → RSA 签名
    → 返回 {data, key, iv, nonce, timestamp, signature} Map
  → ResponseModifyAdvice (@Order HIGHEST_PRECEDENCE + 1)
    → 包装为 R.success(加密Map)
    → 最终: {code:200, msg:"success", data: {data:"...", key:"...", ...}}
```

## 请求处理流程

```
HTTP 请求到达
  → MessageConverter 准备反序列化
  → RequestDecryptAdvice.beforeBodyRead()
    → 检查是否为加密请求（含 data + key + iv 字段）
    → RSA 验签 → RSA 私钥解密 AES 密钥 → AES-GCM 解密 data
    → 返回解密后的 body 供后续反序列化
  → Controller 收到正常的 DTO
```

## 配置

密钥存储在 `sys_config` 数据库表，通过后端 API 管理，前端按需获取：

```
sys_config 表：
├── crypto.enabled              = true/false（加解密总开关）
├── crypto.server.public-key   = 服务端公钥（前端加密请求用）
├── crypto.server.private-key  = 服务端私钥（后端解密请求用）
├── crypto.client.public-key   = 客户端公钥（后端加密响应用）
└── crypto.client-private-key  = 客户端私钥（前端解密响应用，通过 API 获取）
```

前端环境变量：

```env
# spectra-ui .env
VITE_CRYPTO_ENABLED=true
```

## 方案二：密钥迁移至 sys_config（已实现）

> 状态：✅ 已实现 — 2026-07-13
>
> 目标：将 RSA 密钥从 `application.yml` 硬编码迁移至 `sys_config` 表动态管理，支持运行时热更新。

### 背景

方案一存在的安全问题：
- 前端 `.env` 中硬编码 RSA 私钥，通过 `VITE_` 前缀编译进 JS bundle，DevTools 可直接获取
- 密钥轮换需改 yml + 重启服务 + 重新构建前端
- 前后端共享同一对密钥，签名无身份证明价值

方案二改进：
- 后端密钥存入 `sys_config` 表，前端通过 API 按需获取
- 前端不持有持久化私钥，仅在内存中缓存，刷新即失效
- 密钥轮换通过系统配置管理页面即时生效，无需重启

### sys_config 密钥命名

| key | value 示例 | 用途 | API 暴露 |
|---|---|---|---|
| `crypto.enabled` | `true` | 加解密总开关 | `GET /system/crypto/config` |
| `crypto.server.public-key` | `MIIBIjAN...` | 服务端公钥（前端加密请求） | `GET /system/crypto/config` |
| `crypto.server.private-key` | `MIIEvQIB...` | 服务端私钥（后端解密请求） | 不暴露 |
| `crypto.client.public-key` | `MIIBIjAN...` | 客户端公钥（后端加密响应） | 不暴露 |
| `crypto.client.private-key` | `MIIEvQIB...` | 客户端私钥（前端解密响应） | `GET /system/crypto/keypair/client-private` |

### 架构对比

| 方面 | 方案一（yml 硬编码） | 方案二（sys_config 动态） |
|---|---|---|
| 私钥存储 | 前端 JS bundle 明文 | 仅浏览器内存 |
| 密钥轮换 | 改 yml + 重启 + 重构建 | 系统配置页面修改 → 即时生效 |
| 加密开关 | yml 中 `spectra.system.sm.enabled` | `sys_config.crypto.enabled` |
| 服务端私钥 | yml 明文 | sys_config（数据库级别权限隔离） |
| 性能 | 启动时一次性解析 | 启动加载到内存，请求时零开销 |

### CryptoKeyManager 设计

**位置**：`spectra-framework/configure/mvc/crypto/CryptoKeyManager.java`

**核心设计**：用 `JdbcTemplate` 直读 `sys_config` 表（避免依赖 `spectra-core` 导致循环引用），密钥对象用不可变 wrapper + `volatile` 原子替换保证线程安全。

```
CryptoKeyManager
├── @PostConstruct init() → 从 sys_config 加载密钥到内存
├── isEnabled() → 检查 crypto.enabled + 密钥完整性
├── getServerPublicKey() → 返回内存缓存的 PublicKey
├── getServerPrivateKey() → 返回内存缓存的 PrivateKey
├── getClientPublicKey() → 返回内存缓存的 PublicKey
├── getClientPrivateKey() → 返回内存缓存的 PrivateKey
├── getServerPublicKeyBase64() → 返回 Base64 字符串
├── getClientPrivateKeyBase64() → 返回 Base64 字符串
└── refresh() → 重新从 sys_config 加载
```

**线程安全方案**：

```java
private record CryptoKeys(
    PublicKey serverPub, PrivateKey serverPri,
    PublicKey clientPub, PrivateKey clientPri
) {}
private volatile CryptoKeys keys;  // 原子替换，无部分更新窗口
```

**降级策略**：
- sys_config 无数据 → keys 为 null → `isEnabled()` 返回 false → 加解密透传
- DB 连接失败 → 静默失败，不阻塞启动

### 密钥数据流

```
启动时：
  sys_config (DB) → CryptoKeyManager.init() → 内存缓存

请求时：
  ResponseEncryptAdvice → cryptoKeyManager.getClientPublicKey() → 内存（微秒级）
  RequestDecryptAdvice → cryptoKeyManager.getServerPrivateKey() → 内存（微秒级）

密钥轮换：
  管理员修改 sys_config → CryptoController → cryptoKeyManager.refresh()
    → 清空旧内存 → 从 sys_config 重载 → 原子替换
```

### CryptoController API 变更

| 端点 | 方法 | 认证 | 说明 |
|---|---|---|---|
| `GET /system/crypto/config` | 新增 | 无 | 返回 `{enabled, serverPublicKey}`，前端初始化调用 |
| `GET /system/crypto/keypair/server-public` | 保留 | 无 | 返回服务端公钥 |
| `GET /system/crypto/keypair/client-private` | 新增 | 需登录 | 返回客户端私钥 |
| `POST /system/crypto/keypair/generate` | 修改 | `ROLE_DEV_OPS` | 生成两对密钥 → 写入 sys_config → refresh |
| `POST /system/crypto/keypair/refresh` | 新增 | `ROLE_DEV_OPS` | 手动重新加载密钥到内存 |

### 前端初始化时序

```
应用启动
  → GET /system/crypto/config → { enabled: true, serverPublicKey: "..." }
    → enabled=false → 全局跳过加解密
    → enabled=true → 缓存 serverPublicKey → 后续请求加密可用

用户登录
  → POST /auth/login  (@Encrypt(false)，明文响应)
  → GET /system/crypto/keypair/client-private (带 Token)
    → 缓存 clientPrivateKey → 后续响应解密可用
```

### 风险分析

| 风险等级 | 问题 | 处理方式 |
|---|---|---|
| 高 | 循环依赖（CryptoKeyManager 需读 sys_config） | `JdbcTemplate` 直读，不依赖 spectra-core |
| 高 | 启动时 sys_config 无数据 | 静默降级，`isEnabled()` 返回 false |
| 中 | 线程安全（refresh 期间部分更新） | 不可变 wrapper + volatile 原子替换 |
| 中 | `enabled=true` 但密钥缺失 | `isEnabled()` 附加密钥完整性检查 |
| 低 | 密钥变更竞态（请求加密用旧 key，后端解密用新 key） | v1 接受，不做处理 |
| 低 | 前端登录前 POST 响应加密 | 登录前端点已豁免，不受影响 |
